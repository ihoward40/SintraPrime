Architecture Overview

SintraPrime is built as a multi-service architecture where every component operates under a unified governance layer. This design ensures that no AI agent action can bypass policy enforcement, receipt generation, or audit logging.

Service Architectureâ€‹

SintraPrime consists of five primary services, each containerized for production deployment:

Airlock Server (ManusLite Gateway)â€‹

The Airlock is the single entry point for all external payloads entering SintraPrime. It provides:

  • HMAC verification â€” Every incoming request is verified against a shared secret
  • Rate limiting â€” Configurable per-IP request throttling
  • Payload validation â€” Schema validation before forwarding to the Brain
  • Security hardening â€” Helmet headers, CORS controls, and request sanitization

Brain Service (Core Orchestrator)â€‹

The Brain is the heart of SintraPrime. It contains:

  • Agent Mode Engine â€” The Validator â†’ Planner â†’ Executor pipeline
  • Workflow Runner â€” YAML/JSON workflow execution engine
  • Governance Engine â€” Policy gates, mode enforcement, and approval workflows
  • Adapter Manager â€” Governed access to all external services
  • Receipt Generator â€” Cryptographic receipt creation and chain management

FastAPI Serviceâ€‹

The Python-based FastAPI service handles:

  • ML/AI model integrations â€” Advanced reasoning and analysis
  • DeepThink analysis â€” Deterministic analysis with cryptographic verification
  • Data processing â€” Complex data transformations and aggregations

WebApp (Operator Dashboard)â€‹

The web application provides:

  • System monitoring â€” Real-time service health and metrics
  • Receipt browser â€” Search and verify operation receipts
  • Task management â€” Submit, monitor, and approve tasks
  • Evidence viewer â€” Browse and export evidence artifacts

MySQL Databaseâ€‹

Persistent storage for:

  • Receipt ledger â€” Immutable operation records
  • Evidence artifacts â€” Processed evidence with metadata
  • System state â€” Configuration, agent registry, and task queues
  • Audit logs â€” Comprehensive system activity logs

Request Flowâ€‹

Every operation in SintraPrime follows this flow:

  1. Ingress â€” External request arrives at the Airlock Server
  2. Verification â€” HMAC signature is verified, payload is validated
  3. Governance Check â€” The governance engine checks mode, policy gates, and spending limits
  4. Planning â€” The Planner generates an execution plan with required adapters
  5. Approval â€” If required, the task awaits human approval
  6. Execution â€” The Executor runs the plan through governed adapters
  7. Receipt Generation â€” A cryptographic receipt is generated with SHA-256 hash and Ed25519 signature
  8. Chain Update â€” The receipt is chained to the previous receipt via hash linking
  9. Response â€” Results are returned to the caller with receipt reference

Design Philosophy

SintraPrime follows a fail-closed design. If any governance check fails â€” missing policy, invalid mode, exceeded spending limit â€” the operation is rejected and a denial receipt is generated. The system never defaults to allowing an operation.
